# Phase 5 — audit & challenge (avant application)

La Phase 5 du plan (« Producteur d'export + rôle éditeur ») couvre **US-1.4** (FX :
exporter une session jouée en compte rendu publiable sur Suddenly) et **US-2.2**
(utilisateur Suddenly : publier son récit). Comme la Phase 3, elle est
**sous-spécifiée** sur sa couture aval. On l'audite avant de câbler quoi que ce soit.

## Rappel de l'invariant (US-1.4)

> L'export ne franchit la membrane qu'**après résolution** : les secrets non
> révélés sont soit **résolus**, soit **retirés**. Le mur tombe sur le résolu,
> jamais sur le caché vivant.

C'est un **deuxième mur**, de même gravité que celui de `packet.rs` — mais inversé :
le premier interdit au secret de **sortir** vers le narrateur pendant le jeu ; la
membrane décide ce qui peut **sortir** vers Suddenly *après* le jeu.

## Constats d'audit

### 1. La couture CN↔Suddenly est « à spécifier » (comme le Hub en Phase 3)
Le versant aval (upload du compte rendu, « blob → base ») appartient à **Suddenly**,
pas à CN. L'endpoint réel n'existe pas. **Implémentable côté CN** : le *producteur*
(compte rendu + membrane), le *rôle éditeur* (UI de revue/résolution), et un
**`Publisher` stub** abstrait derrière une interface (comme `HttpNarrator`).

### 2. TROU CENTRAL : la prose de la session n'est jamais persistée
Vérifié dans le code :
- `engine.resolve` renvoie `Outcome::Commit { candidat, … }` — la prose est
  **consommée par l'hôte**, jamais stockée.
- `World` (donc le `snapshot`) ne contient **aucun transcript**.
- la `trace` dev (`TourTrace`) est un `Vec` séparé, **perdu au snapshot**.

Conséquence : **une session reprise a perdu tout son historique de prose** (seul
`savoir_joueur` survit). Or l'export *est* ce transcript. → Il faut **journaliser**
les tours commités dans le `World` (donc dans le snapshot). Bénéfice collatéral :
la reprise de session retrouve enfin son historique (utile bien au-delà de l'export).

### 3. Sémantique de « résolu » : le secret n'est JAMAIS révélé en jeu
Le directeur ne lâche que des `revealable` sous budget — **jamais** le
`secret_reponse`. Donc à l'export, le secret est **par défaut non résolu** et
l'éditeur **doit trancher** par secret : `Révéler(texte)` ou `Retirer`. Le
producteur **refuse** de produire tant qu'un secret est indécis (ni l'un ni l'autre).
C'est exactement « résolus, soit retirés ».

### 4. La membrane doit être Rust-autoritaire, canon-aware et paranoïaque
- **Autorité** : le canon vit en Rust ; la décision « ce fait peut sortir » doit
  être au même endroit que le canon, pas seulement en TS. → `Engine::export`.
- **Paranoïa (défense en profondeur)** : même si la prose journalisée a passé le
  verifier (donc sans `jetons_fuite`), l'export **re-scanne** chaque échange et
  **refuse** si un jeton de fuite non résolu apparaît. Un mur ne fait jamais
  confiance à l'étage du dessous.
- **Sortie close** : le compte rendu est un type fermé (comme le paquet) — il ne
  contient QUE scène publique + échanges + faits appris + résolutions explicites.

### 5. Rôle éditeur = produit CN, publish = abstrait
L'éditeur (US-1.4) revoit le compte rendu, édite la prose, tranche chaque secret.
C'est une **UI CN**. La transmission à Suddenly est un `Publisher` **stub** (la
couture restant à spécifier). Le même artefact sert FX et l'utilisateur Suddenly.

### 6. Compatibilité snapshot
Le champ `journal` ajouté au `World` doit être `#[serde(default)]` : les snapshots
Phase 3/4 (sans journal) doivent continuer à se charger.

## Ce qu'on implémentera (côté CN, éprouvable)

| Brique | Rôle |
| --- | --- |
| `World.journal` (Rust, serde default) | transcript persistant : action + prose + faits par tour commité |
| `Engine::export(resolutions)` (Rust) | **membrane** : refuse si secret indécis ou fuite détectée ; sinon `CompteRendu` clos |
| `CompteRendu` (Rust, Serialize) | type fermé : scène publique, échanges, faits appris, résolutions |
| `WasmEngine.export(json)` | façade JS → `Result<CompteRendu JSON, erreur string>` |
| `narrative/export.ts` + types | miroir TS du compte rendu + interface `Publisher` + stub |
| UI éditeur (Astro) | revue, édition de prose, décision résolus/retirés par secret, aperçu |

## Bloqueurs non testables ici
- L'upload Suddenly réel (auth, format « base ») — abstrait derrière `Publisher`.
- Le pont **vault** (`rpg-writer`) : exporter le compte rendu *vers le vault*
  (`campagnes/<c>/mj/`) est une variante du `Publisher` — naturellement Phase 5+.

## Décisions à trancher avant code
1. **Où vit la membrane** : `Engine::export` Rust-autoritaire (recommandé) vs
   producteur TS seul.
2. **Format du compte rendu** : Markdown lisible (éditeur/publi) vs blob structuré
   (« blob → base ») vs **les deux** (JSON canonique + rendu Markdown — recommandé).

## Recommandation
**Go** pour la part CN, à condition de traiter le trou #2 (journal persistant) en
premier — sans lui, l'export est vide après reprise. La membrane (#4) est l'invariant
critique : Rust-autoritaire, canon-aware, paranoïaque, type de sortie fermé.
