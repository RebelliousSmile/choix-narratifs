# Phase 3 — audit & épreuve (avant application)

La Phase 3 du plan (« Island Astro `client:only` + IndexedDB + reprise de session »)
était **sous-spécifiée**. Plutôt que de la câbler à l'aveugle, on l'a auditée puis
éprouvée par des tests sur tout ce qui ne dépend **pas** du navigateur/WASM réel.

## Constats d'audit

1. **« Island `client:only` » est imprécis.** `client:only` cible les composants de
   _framework_ (React/Vue/Svelte). Le projet n'en utilise pas : son pattern d'island
   est un **`<script>` Astro** important des modules `src/scripts/` (cf.
   `plan-de-classe.astro`). Le câblage réel sera un `<script>` qui fait un
   `import()` dynamique du glue WASM — pas un composant `client:only`.

2. **IndexedDB vs `storage.ts` (localStorage) — faux conflit.** Le moteur narratif
   est un **produit séparé** des outils v1 (cartes, plan de classe, sélecteur de
   mues, parallaxe). Il **ne réutilise pas** `storage.ts` ni ses conventions. Choix
   d'IndexedDB justifié : le snapshot est un `Uint8Array` opaque → stockage binaire
   natif (structured clone), asynchrone, « seul écrivain en solo ». localStorage
   imposerait du base64 gonflé. → espace dédié `src/scripts/narrative/`.

3. **Dépendances de test manquantes.** jsdom n'implémente pas IndexedDB → ajout de
   `fake-indexeddb` (dev) pour éprouver le **vrai** chemin de persistance.

4. **Bloqueurs non testables ici (réseau/navigateur).** La génération du glue JS
   (`wasm-pack` → `pkg/`) et l'exécution du `.wasm` exigent un navigateur ; le relais
   `/narrate` exige le Hub + auth. Ces points sont **abstraits derrière des
   interfaces** et restent à câbler/éprouver en environnement réel.

## Ce qui a été éprouvé (testé, vert)

Briques dans `src/scripts/narrative/`, découplées par interfaces :

| Brique | Rôle | Épreuve |
| --- | --- | --- |
| `types.ts` | miroir TS du contrat émis par cn-wasm | `contract.test.ts` parse de **vraies** fixtures Rust |
| `engine.ts` | interface `NarrativeEngine` (ce que `WasmEngine` implémente) | doubles sans WASM |
| `narrator.ts` | interface `Narrator` + `HttpNarrator` + `assertCanonShape` | — |
| `snapshot-store.ts` | `IdbSnapshotStore` (IndexedDB) + `MemorySnapshotStore` | `snapshot-store.test.ts` (fake-indexeddb) |
| `session.ts` | la boucle hôte `runTurn` + `resume` | `session.test.ts` |

15 tests narratifs (3 contrat + 6 boucle + 6 store), **129 au total**, `pnpm check` 0 erreur.

Scénarios prouvés : fuyard écarté → commit ; tous invalides → resample invisible
→ commit ; épuisement après `maxResamples` ; même paquet re-soumis au resample
(beat ouvert) ; snapshot persisté après commit ; reprise du savoir depuis snapshot.
Alignement TS ↔ Rust vérifié sur fixtures réelles (clé `move`, enums snake_case,
`Outcome`/`Rejet` discriminés, aucun secret dans le paquet).

## Défaut réel trouvé en éprouvant

**Snapshot tronqué / mal typé au rechargement.** Le structured clone d'IndexedDB
restitue un `Uint8Array` d'un **autre realm** : `instanceof Uint8Array` est **faux**
(observé sous fake-indexeddb, possible selon les moteurs). La 1re version de
`IdbSnapshotStore.load` levait « format inattendu ». Corrigé en testant
`ArrayBuffer.isView` et en recopiant la fenêtre `[byteOffset, +byteLength]` (gère
aussi une vue sur la mémoire WASM). Sans l'épreuve, ce bug serait apparu en prod.

- Câbler un `<script>` Astro : `import init, { WasmEngine }`, instancier, brancher
  `runTurn` sur un `HttpNarrator` réel + `IdbSnapshotStore`, et l'UI de surface de jeu.

- `id` de session + politique de reprise au montage de la page.

et l'API colle au contrat Rust. L'application se réduit à (a) générer `pkg/`,
(b) un adaptateur `WasmEngine → NarrativeEngine` (quasi 1:1), (c) la page + l'UI.
À faire dans un environnement avec navigateur/`wasm-pack`, pas dans cette session.
